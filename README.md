# sysgo

`sysgo` ingests a **SysML v2 model**, normalizes it into a **DDD/hexagonal
intermediate representation**, and emits a **configurable Go project scaffold**
whose layout follows the canonical Evans / Cockburn / Martin architecture. Its
UX (config file, templates, overlays) is modeled directly on
[`oapi-codegen`](https://github.com/oapi-codegen/oapi-codegen).

See [`SPEC.md`](./SPEC.md) for the full design.

## Install

```bash
go install github.com/gaarutyunov/sysgo/cmd/sysgo@latest
```

## Quick start

```bash
sysgo init                       # scaffold sysgo.yaml + overlay.yaml
sysgo validate -c sysgo.yaml     # load model + report mapping diagnostics (writes nothing)
sysgo generate -c sysgo.yaml     # run the pipeline and emit the scaffold
sysgo version
```

Try it against the bundled example:

```bash
sysgo generate -c examples/order/sysgo.yaml --out ./out
```

## From SysML source to Go

sysgo consumes the **SysML v2 API JSON**. You author models in the SysML v2
**textual notation** (`.sysml`) and transform them to that JSON with real SysML
tooling first ‚Äî the JSON is an intermediate, not something you hand-write:

```bash
# .sysml ‚îÄ‚îÄ[ OMG SysML v2 Pilot serializer ]‚îÄ‚îÄ‚ñ∂ model.json ‚îÄ‚îÄ[ sysgo ]‚îÄ‚îÄ‚ñ∂ Go
scripts/sysml2json.sh examples/order/OrderContext.sysml examples/order/model.json
sysgo generate -c examples/order/sysgo.yaml --out ./out
```

`scripts/sysml2json.sh` (also `make model`) drives the OMG **SysML v2 Pilot
Implementation** serializer (`org.omg.sysml.xtext.util.SysML2JSON`), which
parses the textual notation against the standard libraries and emits the
`{payload, ‚Ä¶}` element envelope. It requires Java 17+ and `curl`, and caches the
pilot distribution under `~/.cache/sysml`. The example ships both the real
source ([`examples/order/OrderContext.sysml`](./examples/order/OrderContext.sysml))
and the tool-produced intermediate (`examples/order/model.json`); CI regenerates
the JSON from source and asserts the generated Go is unchanged.

Already have JSON from the SysML v2 REST API or another export? Point
`source.file` (or `source.api`) at it directly ‚Äî element @ids differ between
tool runs but sysgo's output does not depend on them.

## How it works

```
SysML v2 ‚îÄ‚îÄ‚ñ∫ Loader ‚îÄ‚îÄ‚ñ∫ Overlay Engine ‚îÄ‚îÄ‚ñ∫ IR Builder ‚îÄ‚îÄ‚ñ∫ Renderer ‚îÄ‚îÄ‚ñ∫ Emitter ‚îÄ‚îÄ‚ñ∫ Go project
 (API/JSON)  (graph)     (JSONPath)        (DDD model)   (templates)   (gofmt)
```

1. **Loader** ‚Äî reads the SysML v2 element graph from the REST API
   (`source.api`) or an exported JSON file (`source.file`) and resolves
   memberships into a navigable tree.
2. **Overlay Engine** ‚Äî applies OpenAPI-Overlay-style `update`/`remove`/`copy`
   actions to the raw model JSON before anything is interpreted.
3. **IR Builder** ‚Äî applies heuristics + metadata to produce the DDD IR
   (entities, value objects, ports, use cases, events).
4. **Renderer** ‚Äî executes embedded (or user-overridden) `text/template`s.
5. **Emitter** ‚Äî writes files, runs `gofmt`, applies the generated /
   scaffold-once policy, and prunes stale generated files.

## Generated layout

The output is **bounded-context-first** and obeys the **Dependency Rule** ‚Äî
every cross-region import points inward:

```
cmd/<context>d/main.go                       # composition root (scaffold-once)
internal/<context>/
  domain/        # entities, value objects, events, factories, domain services
  app/
    port/in/     # driving ports (use-case interfaces) + DTOs
    port/out/    # driven ports (repository / gateway interfaces)
    usecase/     # interactors (scaffold-once)
  adapter/
    in/http/     # driving adapters     (scaffold-once)
    out/repository, out/gateway          (scaffold-once)
.go-arch-lint.yml                            # arch-lint ruleset (optional)
```

- **Generated files** carry `// Code generated by sysgo; DO NOT EDIT.` and are
  always overwritten.
- **Scaffold-once files** (adapters, interactors, `main.go`) are written only if
  absent and never overwritten.

## Configuration

`sysgo.yaml` is validated against a JSON Schema
([`schema/sysgo.schema.json`](./schema/sysgo.schema.json)). The key sections:

| Section | Purpose |
|---|---|
| `module` | Go module path |
| `source` | `api` (base-url/project/commit) **or** `file` |
| `overlay.path` | overlay document applied before IR build |
| `generate.*` | which artifacts to emit (`adapters: off\|scaffold\|full`) |
| `ports.*` | port directories; `repository-in-domain` |
| `layout.*` | region ‚Üí directory + package (`{context}` interpolated) |
| `type-mapping` / `import-mapping` / `additional-imports` | Go type/import control |
| `output-options.*` | user templates, `skip-fmt`, `skip-prune`, marker |

Precedence (most specific wins): element metadata ‚Üí overlay-injected keys ‚Üí
`type-mapping`/`layout` rules ‚Üí built-in defaults.

## Overlays

Overlays patch the model JSON without editing the source (OpenAPI-Overlay
grammar, plus `byName:`/`byType:` selector sugar):

```yaml
overlay: 1.0.0
info: { title: Orders Go overlay, version: 1.0.0 }
actions:
  - target: $[?(@.declaredName=='price')]
    update: { x-go-type: "money.Money", x-go-type-import: "github.com/acme/money" }
  - target: $[?(@.declaredName=='Order' && @['@type']=='PartDefinition')]
    update: { x-ddd-stereotype: "aggregate" }
  - target: $[?(@['@type']=='PartDefinition' && @.x-internal==true)]
    remove: true
```

## Templates

The default `text/template` set is embedded and overridable per file
(`output-options.user-templates` or `--templates <dir>`, same-name semantics).
Helpers: `goName`, `exported`, `unexported`, `comment`, `zeroValue`,
`receiver`, `sig`, and a `header` partial.
